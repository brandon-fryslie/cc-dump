# Events

> Status: draft
> Last verified against: not yet

## Overview

Claude Code communicates with the Anthropic API over HTTP, but none of that traffic is visible to the user. The event system exists to make every API call observable: what was sent, what is streaming back, and when it finishes. Without it, cc-dump would have no way to feed its TUI, recording, or analytics subsystems — users would be staring at a blank screen while Claude Code works.

From the user's perspective, every Claude Code API call appears as a conversation turn in the TUI. The event system is the machinery that makes this happen: the proxy intercepts HTTP traffic, parses it into typed events, and distributes those events to all consumers (TUI, HAR recorder, analytics store) through a fan-out router. The event types define the contract between "what the proxy saw" and "what the rest of the system does with it."

## Event Hierarchy

There are two distinct event hierarchies: **SSE events** (raw streaming protocol) and **pipeline events** (the system-wide event bus).

### SSE Events

SSE events represent individual Server-Sent Events from the Anthropic streaming API. They are parsed at the proxy boundary and exist only transiently — they feed the response assembler and progress extraction, but are never placed on the pipeline event bus directly.

Base class: `SSEEvent` (frozen dataclass, no fields).

| Type | Fields | Meaning |
|------|--------|---------|
| `MessageStartEvent` | `message: MessageInfo` | Stream opened. Carries message ID, role, model, initial usage. |
| `TextBlockStartEvent` | `index: int` | A text content block began at this index. |
| `ToolUseBlockStartEvent` | `index: int`, `id: str`, `name: str` | A tool_use content block began. `name` is the tool name (e.g., "Read", "Task"). |
| `TextDeltaEvent` | `index: int`, `text: str` | Incremental text content for block at `index`. |
| `InputJsonDeltaEvent` | `index: int`, `partial_json: str` | Incremental JSON fragment for a tool_use block's `input` field. |
| `ContentBlockStopEvent` | `index: int` | Content block at `index` is complete. |
| `MessageDeltaEvent` | `stop_reason: StopReason`, `stop_sequence: str`, `output_tokens: int` | Message-level metadata delta. Carries final stop reason and output token count. |
| `MessageStopEvent` | _(none)_ | Message stream is complete. |

**Value types used by SSE events:**

`MessageInfo` (frozen dataclass): `id: str`, `role: MessageRole`, `model: str`, `usage: Usage`

`Usage` (frozen dataclass): `input_tokens: int = 0`, `output_tokens: int = 0`, `cache_read_input_tokens: int = 0`, `cache_creation_input_tokens: int = 0`

`MessageRole` (enum): `USER`, `ASSISTANT`

`StopReason` (enum): `NONE("")`, `END_TURN`, `MAX_TOKENS`, `STOP_SEQUENCE`, `TOOL_USE`, `STOP` (OpenAI), `LENGTH` (OpenAI), `TOOL_CALLS` (OpenAI), `CONTENT_FILTER` (OpenAI)

### Pipeline Events

Pipeline events are the system-wide event bus. Every event that any consumer (TUI, HAR recorder, analytics) sees is a `PipelineEvent` subclass. All pipeline events share a common envelope.

Base class: `PipelineEvent` (frozen dataclass).

**Envelope fields** (present on every pipeline event):

| Field | Type | Default | Meaning |
|-------|------|---------|---------|
| `kind` | `PipelineEventKind` | _(set by subclass, not caller)_ | Discriminator enum. The class IS the type — `kind` exists for dispatch tables. |
| `request_id` | `str` | `""` | Groups all events belonging to one API call. UUID hex, generated by `new_request_id()`. |
| `seq` | `int` | `0` | Monotonically increasing within a `request_id`. Defines ordering. |
| `recv_ns` | `int` | `0` | `time.monotonic_ns()` timestamp of when the event was created. |
| `provider` | `str` | `"anthropic"` | Which API provider this event belongs to (e.g., `"anthropic"`, `"openai"`). |

**Pipeline event types:**

| Type | Kind | Additional Fields | Meaning |
|------|------|-------------------|---------|
| `RequestHeadersEvent` | `REQUEST_HEADERS` | `headers: dict[str, str]` | HTTP request headers (sensitive headers stripped). |
| `RequestBodyEvent` | `REQUEST` | `body: dict[str, object]` | Parsed JSON request body. |
| `ResponseHeadersEvent` | `RESPONSE_HEADERS` | `status_code: int`, `headers: dict[str, str]` | HTTP response headers (sensitive headers stripped). |
| `ResponseSSEEvent` | `RESPONSE_EVENT` | `sse_event: SSEEvent` | Dead code — defined but never constructed by the proxy. |
| `ResponseProgressEvent` | `RESPONSE_PROGRESS` | `delta_text: str`, `model: str`, `input_tokens: int\|None`, `cache_read_input_tokens: int\|None`, `cache_creation_input_tokens: int\|None`, `output_tokens: int\|None`, `stop_reason: str`, `task_tool_use_id: str` | Transport-normalized streaming hint for live UX. Not authoritative — use `ResponseCompleteEvent` for final data. All fields default to empty/None. |
| `ResponseNonStreamingEvent` | `RESPONSE_NON_STREAMING` | `status_code: int`, `headers: dict[str, str]`, `body: dict` | Dead code — the proxy emits `ResponseCompleteEvent` for both streaming and non-streaming. |
| `ResponseCompleteEvent` | `RESPONSE_COMPLETE` | `body: dict` | Fully assembled response. For streaming responses, reconstructed from SSE fragments by `ResponseAssembler`. For non-streaming, the parsed response body directly. Same shape as a `stream=false` API response. |
| `ResponseDoneEvent` | `RESPONSE_DONE` | _(none)_ | Terminal event — response stream is fully consumed. Emitted after `ResponseCompleteEvent`. |
| `ErrorEvent` | `ERROR` | `code: int`, `reason: str` | HTTP error from upstream (e.g., 429, 500). |
| `ProxyErrorEvent` | `PROXY_ERROR` | `error: str` | Proxy-level failure (connection refused, timeout, etc.). |
| `LogEvent` | `LOG` | `method: str`, `path: str`, `status: str` | HTTP access log entry. Emitted for ALL HTTP requests (not just API requests) via the `log_message()` override. Non-API traffic (health checks, etc.) still produces `LogEvent` even though no request/response pipeline events are emitted. |

## Event Lifecycle of a Single Streaming API Call

This is the canonical sequence for a typical Claude Code request that streams its response.

### 1. Request Phase (proxy thread)

The proxy receives an HTTP POST from Claude Code, parses the JSON body, and emits two events into the shared `event_queue`:

```
RequestHeadersEvent  (seq=0)   — filtered HTTP headers
RequestBodyEvent     (seq=1)   — parsed JSON body (messages, tools, system, etc.)
```

Both events share the same `request_id`. Headers are emitted first so the TUI can display them before the potentially large body.

**Excluded headers** (never emitted): `authorization`, `x-api-key`, `cookie`, `set-cookie`, `host`, `content-length`, `transfer-encoding`.

**Gate condition:** Request events are only emitted for API-path requests with a valid JSON body. Non-API traffic (health checks, token counting endpoints) is forwarded to the client but produces no pipeline events (though `LogEvent` is still emitted for all HTTP requests via `log_message()`).

### 2. Upstream Forwarding

The proxy forwards the request to the Anthropic API. If the request pipeline has transforms, they modify the body/URL before forwarding. If an interceptor short-circuits (e.g., the sentinel `$$` command), a synthetic SSE response is generated instead.

### 3. Response Headers

When the upstream response arrives:

```
ResponseHeadersEvent (seq=0)   — status code + filtered response headers
```

The proxy checks `Content-Type: text/event-stream` to determine streaming vs. non-streaming path.

### 4a. Streaming Response (typical path)

The SSE stream is processed by `_fan_out_sse`, which drives three sinks in parallel:

| Sink | Role |
|------|------|
| `ClientSink` | Forwards raw bytes to Claude Code (transparent proxying). |
| `EventQueueSink` | Parses SSE events, extracts progress hints, emits `ResponseProgressEvent`. |
| `ResponseAssembler` | Accumulates SSE fragments, reconstructs complete response. |

During streaming, `ResponseProgressEvent` events are emitted for SSE events that carry user-visible information:

| SSE Event | Progress Fields Extracted |
|-----------|-------------------------|
| `message_start` | `model`, `input_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens` |
| `content_block_delta` (text) | `delta_text` |
| `message_delta` | `output_tokens`, `stop_reason` |
| `content_block_start` (tool_use, name="Task") | `task_tool_use_id` |

Each `ResponseProgressEvent` increments `seq`.

Not all SSE events produce progress events. `content_block_start` (text), `content_block_stop`, `content_block_delta` (input_json), and `message_stop` are silently consumed by the assembler but emit no progress hint.

### 4b. Non-Streaming Response (alternative path)

For non-streaming responses, the proxy reads the entire body, then emits:

```
ResponseHeadersEvent    (seq=0)  — status + headers
ResponseCompleteEvent   (seq=1)  — parsed response body
```

No `ResponseProgressEvent` or `ResponseDoneEvent` is emitted for this path. Confirmed: the non-streaming path does NOT emit `ResponseDoneEvent`. Only `ResponseHeadersEvent` and `ResponseCompleteEvent`.

### 5. Stream Completion

After the SSE stream ends (or the assembler finishes for non-streaming), the proxy emits the finalization events:

```
ResponseCompleteEvent  (seq=N)    — fully reconstructed message
ResponseDoneEvent      (seq=N+1)  — terminal signal
```

`ResponseCompleteEvent.body` has the same shape as a `stream=false` Anthropic Messages API response: `{id, type, role, content, model, stop_reason, stop_sequence, usage}`.

`ResponseDoneEvent` is always the last event for a request_id in the streaming path. It signals that all processing for this API call is finished.

### 6. Error Paths

**HTTP error from upstream** (e.g., 429 rate limit, 500 server error):

```
ErrorEvent (code=429, reason="Too Many Requests")
```

No `ResponseDoneEvent` follows. The error is terminal for this request_id.

**Proxy-level failure** (connection refused, timeout, DNS failure):

```
ProxyErrorEvent (error="Connection refused")
```

No `ResponseDoneEvent` follows.

In both error cases, the HTTP error response is still forwarded to Claude Code. Pipeline events are only emitted if request events were emitted for this request_id.

## Complete Event Sequence Example

A typical Claude Code turn where the assistant responds with text:

```
                                         seq   recv_ns
RequestHeadersEvent  {headers: {...}}     0     t0
RequestBodyEvent     {body: {...}}        1     t1
ResponseHeadersEvent {200, headers}       0     t2
ResponseProgressEvent {model: "claude-opus-4-20250514", input_tokens: 12400, ...}
                                          1     t3
ResponseProgressEvent {delta_text: "I"}   2     t4
ResponseProgressEvent {delta_text: "'ll"} 3     t5
ResponseProgressEvent {delta_text: " read"}
                                          4     t6
...
ResponseProgressEvent {output_tokens: 847, stop_reason: "end_turn"}
                                          N-2   t_end-2
ResponseCompleteEvent {body: <full message>}
                                          N-1   t_end-1
ResponseDoneEvent                         N     t_end
```

All events share the same `request_id`. There are two independent `seq` counters: request-side (starting at 0 for `RequestHeadersEvent`, incrementing for `RequestBodyEvent`) and response-side (starting at 0 for `ResponseHeadersEvent`, incrementing through progress and completion events). They are not globally monotonic across the request/response boundary.

## Replay Path

HAR replay (`har_replayer.py`) produces the same pipeline event types as the live proxy, but with a simplified sequence since there is no streaming:

```
RequestHeadersEvent   (seq=0)
RequestBodyEvent      (seq=1)
ResponseHeadersEvent  (seq=2)
ResponseCompleteEvent (seq=3)
```

No `ResponseProgressEvent` or `ResponseDoneEvent` is emitted during replay. The replay path uses `event_envelope()` to generate `recv_ns` timestamps at replay time (not original capture time).

This means consumers must handle both "streaming with progress" and "direct complete" event sequences. The shared finalization path is `ResponseCompleteEvent` — it is the authoritative event for response content regardless of transport.

## Event Distribution

### Router Architecture

Events flow through a single `queue.Queue[PipelineEvent]` from producers (proxy, replayer) to the `EventRouter`, which fans out to subscribers on a dedicated daemon thread.

```
Producer(s) → Queue → EventRouter thread → Subscriber 1 (on_event)
                                         → Subscriber 2 (on_event)
                                         → Subscriber N (on_event)
```

### Subscriber Types

| Type | Delivery | Use Case |
|------|----------|----------|
| `QueueSubscriber` | Buffers into its own `queue.Queue` for async consumption | TUI (events are drained on the Textual thread) |
| `DirectSubscriber` | Calls a function inline on the router thread | HAR recorder, analytics store, tmux controller |

### Ordering Guarantees

- Events for a single `request_id` are produced in causal order by the proxy thread. Since the proxy handles one request at a time per connection, request/response events for the same `request_id` are always ordered.
- The router preserves insertion order: events are drained from the source queue in FIFO order and delivered to each subscriber in subscriber-registration order.
- All subscribers see the same event in the same order.
- A subscriber exception does not prevent other subscribers from receiving the event (logged and swallowed by the router).
- `QueueSubscriber` consumers must drain their own queue to maintain ordering; the router does not block on subscriber delivery.

### Concurrent Requests

Multiple API calls can be in flight simultaneously (e.g., Claude Code making parallel tool calls). Events from different `request_id` values may interleave on the shared queue. Consumers must use `request_id` to correlate events belonging to the same API call.

## SSE Parse Boundary

`parse_sse_event(event_type, raw)` is the single enforcer for SSE data validation. It is called at one production boundary only:

1. **`proxy.py`** — live SSE stream parsing

The replayer does NOT call `parse_sse_event` — it constructs events directly from HAR body.

The parse boundary converts raw JSON dicts into typed `SSEEvent` subclasses. Unknown event types raise `ValueError`. The dispatch table (`_SSE_PARSERS`) maps SSE type strings to parser functions:

| SSE type string | Parser output |
|----------------|---------------|
| `"message_start"` | `MessageStartEvent` |
| `"content_block_start"` | `TextBlockStartEvent` or `ToolUseBlockStartEvent` (based on `content_block.type`) |
| `"content_block_delta"` | `TextDeltaEvent` or `InputJsonDeltaEvent` (based on `delta.type`) |
| `"content_block_stop"` | `ContentBlockStopEvent` |
| `"message_delta"` | `MessageDeltaEvent` |
| `"message_stop"` | `MessageStopEvent` |

## Response Assembly

The `ResponseAssembler` (Anthropic) and `OpenAiChatResponseAssembler` (OpenAI) reconstruct complete API responses from SSE fragments. They implement the `StreamSink` protocol (`on_raw`, `on_event`, `on_done`) and participate in `_fan_out_sse` alongside the client and event sinks.

Assembly happens once, at the proxy boundary. The assembler accumulates SSE events during streaming and produces a complete message dict on `on_done()`. This dict is then wrapped in a `ResponseCompleteEvent` and placed on the event queue.

The reconstructed message has the same shape as an Anthropic Messages API response with `stream=false`:

```python
{
    "id": "msg_...",
    "type": "message",
    "role": "assistant",
    "content": [
        {"type": "text", "text": "..."},
        {"type": "tool_use", "id": "toolu_...", "name": "Read", "input": {...}},
    ],
    "model": "claude-opus-4-20250514",
    "stop_reason": "end_turn",
    "stop_sequence": null,
    "usage": {
        "input_tokens": 12400,
        "output_tokens": 847,
        "cache_read_input_tokens": 11000,
        "cache_creation_input_tokens": 0,
    },
}
```

## Multi-Provider Support

The event system supports multiple API providers (Anthropic and OpenAI). The `provider` field on every pipeline event identifies the source. Provider-specific behavior is isolated to:

- **Progress extraction:** `_PROGRESS_EXTRACTORS_BY_FAMILY` dispatches by protocol family (`"anthropic"` or `"openai"`).
- **Response assembly:** `_ASSEMBLER_CLASSES_BY_FAMILY` selects `ResponseAssembler` (Anthropic) or `OpenAiChatResponseAssembler` (OpenAI).
- **SSE parsing:** Anthropic SSE events use typed `parse_sse_event`; OpenAI events are handled by the OpenAI-specific progress extractor directly from raw dicts.

All downstream consumers see the same `PipelineEvent` types regardless of provider. The `ResponseCompleteEvent.body` shape differs by provider (Anthropic Messages format vs. OpenAI Chat Completions format).

## Sentinel Interceptor

The sentinel system (`sentinel.py`) is a request pipeline interceptor that detects `$$` prefixed user messages and short-circuits the API call with a synthetic response. When triggered:

1. The request events (`RequestHeadersEvent`, `RequestBodyEvent`) are emitted normally.
2. Instead of forwarding to upstream, the proxy generates synthetic SSE bytes.
3. The synthetic response flows through the same event emission path: `ResponseHeadersEvent` → `ResponseProgressEvent`(s) → `ResponseCompleteEvent` → `ResponseDoneEvent`.

This ensures the TUI, HAR recorder, and analytics all see a consistent event sequence even for intercepted requests.
