# Events

## Overview

Claude Code communicates with the Anthropic API over HTTP, but none of that traffic is visible to the user. The event system exists to make every API call observable: what was sent, what is streaming back, and when it finishes. Without it, cc-dump would have no way to feed its TUI, recording, or analytics subsystems — users would be staring at a blank screen while Claude Code works.

From the user's perspective, every Claude Code API call appears as a conversation turn in the TUI. The event system is the machinery that makes this happen: the proxy intercepts HTTP traffic, parses it into typed events, and distributes those events to all consumers (TUI, HAR recorder, analytics store) through a fan-out router. The event types define the contract between "what the proxy saw" and "what the rest of the system does with it."

## Event Hierarchy

There are two distinct event hierarchies: **SSE events** (raw streaming protocol) and **pipeline events** (the system-wide event bus).

### SSE Events

SSE events represent individual Server-Sent Events from the Anthropic streaming API. They are parsed at the proxy boundary and exist only transiently — they feed progress extraction and the `sse_progress_payload()` function, but are never placed on the pipeline event bus directly.

Base class: `SSEEvent` (frozen dataclass, no fields).

| Type | Fields | Meaning |
|------|--------|---------|
| `MessageStartEvent` | `message: MessageInfo` | Stream opened. Carries message ID, role, model, initial usage. |
| `TextBlockStartEvent` | `index: int` | A text content block began at this index. |
| `ToolUseBlockStartEvent` | `index: int`, `id: str`, `name: str` | A tool_use content block began. `name` is the tool name (e.g., "Read", "Task"). |
| `TextDeltaEvent` | `index: int`, `text: str` | Incremental text content for block at `index`. |
| `InputJsonDeltaEvent` | `index: int`, `partial_json: str` | Incremental JSON fragment for a tool_use block's `input` field. |
| `ContentBlockStopEvent` | `index: int` | Content block at `index` is complete. |
| `MessageDeltaEvent` | `stop_reason: StopReason`, `stop_sequence: str`, `output_tokens: int` | Message-level metadata delta. Carries final stop reason and output token count. |
| `MessageStopEvent` | _(none)_ | Message stream is complete. |

**Value types used by SSE events:**

`MessageInfo` (frozen dataclass): `id: str`, `role: MessageRole`, `model: str`, `usage: Usage`

`Usage` (frozen dataclass): `input_tokens: int = 0`, `output_tokens: int = 0`, `cache_read_input_tokens: int = 0`, `cache_creation_input_tokens: int = 0`

`MessageRole` (enum): `USER`, `ASSISTANT`

`StopReason` (enum): `NONE("")`, `END_TURN`, `MAX_TOKENS`, `STOP_SEQUENCE`, `TOOL_USE`, `STOP` (OpenAI), `LENGTH` (OpenAI), `TOOL_CALLS` (OpenAI), `CONTENT_FILTER` (OpenAI)

**Note on scope:** SSE events are only used for the Anthropic protocol family. OpenAI SSE events are handled by the OpenAI-specific progress extractor (`_extract_openai_chat_progress`) directly from raw dicts — they are never parsed into typed `SSEEvent` subclasses.

### Pipeline Events

Pipeline events are the system-wide event bus. Every event that any consumer (TUI, HAR recorder, analytics) sees is a `PipelineEvent` subclass. All pipeline events share a common envelope.

Base class: `PipelineEvent` (frozen dataclass).

**Envelope fields** (present on every pipeline event):

| Field | Type | Default | Meaning |
|-------|------|---------|---------|
| `kind` | `PipelineEventKind` | _(set by subclass via `field(init=False)`, not caller)_ | Discriminator enum. The class IS the type — `kind` exists for dispatch tables. |
| `request_id` | `str` | `""` | Groups all events belonging to one API call. UUID hex, generated by `new_request_id()`. |
| `seq` | `int` | `0` | Monotonically increasing within a `request_id`. See "Sequence Number Semantics" below. |
| `recv_ns` | `int` | `0` | `time.monotonic_ns()` timestamp of when the event was created. Generated by `event_envelope()` when not explicitly provided. |
| `provider` | `str` | `"anthropic"` | Which API provider this event belongs to. Known values: `"anthropic"`, `"openai"`, `"copilot"`. |

All envelope fields are `kw_only=True` except `kind`. The `kind` field is `init=False` — it is set by the subclass default, never by the caller.

**Pipeline event types:**

| Type | Kind | Additional Fields | Meaning |
|------|------|-------------------|---------|
| `RequestHeadersEvent` | `REQUEST_HEADERS` | `headers: dict[str, str]` | HTTP request headers (sensitive headers stripped). |
| `RequestBodyEvent` | `REQUEST` | `body: dict[str, object]` | Parsed JSON request body. |
| `ResponseHeadersEvent` | `RESPONSE_HEADERS` | `status_code: int`, `headers: dict[str, str]` | HTTP response headers (sensitive headers stripped). |
| `ResponseSSEEvent` | `RESPONSE_EVENT` | `sse_event: SSEEvent` | **Dead code.** Defined but never constructed. The TUI event handler (`event_handlers.py`) registers a handler that translates it into `ResponseProgressEvent`, but no producer ever creates this event type. Retained for backward compatibility with the handler dispatch table. |
| `ResponseProgressEvent` | `RESPONSE_PROGRESS` | `delta_text: str`, `model: str`, `input_tokens: int\|None`, `cache_read_input_tokens: int\|None`, `cache_creation_input_tokens: int\|None`, `output_tokens: int\|None`, `stop_reason: str`, `task_tool_use_id: str` | Transport-normalized streaming hint for live UX. Not authoritative — use `ResponseCompleteEvent` for final data. All fields default to empty string or None. |
| `ResponseNonStreamingEvent` | `RESPONSE_NON_STREAMING` | `status_code: int`, `headers: dict[str, str]`, `body: dict` | **Dead code.** Defined but never constructed. The TUI event handler registers a handler that normalizes it into the complete-response path, but no producer ever creates this event type. |
| `ResponseCompleteEvent` | `RESPONSE_COMPLETE` | `body: dict` | Fully assembled response. For streaming responses, reconstructed from SSE fragments by `ResponseAssembler` (Anthropic) or `OpenAiChatResponseAssembler` (OpenAI). For non-streaming, the parsed response body directly. |
| `ResponseDoneEvent` | `RESPONSE_DONE` | _(none)_ | Terminal event — response stream is fully consumed. Emitted after `ResponseCompleteEvent`. Emitted for the streaming path and the synthetic interceptor path, but NOT for the non-streaming path. |
| `ErrorEvent` | `ERROR` | `code: int`, `reason: str` | HTTP error from upstream (e.g., 429, 500). |
| `ProxyErrorEvent` | `PROXY_ERROR` | `error: str` | Proxy-level failure (connection refused, timeout, DNS failure, etc.). |
| `LogEvent` | `LOG` | `method: str`, `path: str`, `status: str` | HTTP access log entry. Emitted for ALL HTTP requests (not just API requests) via the `log_message()` override. Non-API traffic (health checks, etc.) still produces `LogEvent` even though no request/response pipeline events are emitted. |

### Sequence Number Semantics

There are two independent `seq` counters per `request_id`:

- **Request-side:** Starts at 0 for `RequestHeadersEvent`, increments to 1 for `RequestBodyEvent`.
- **Response-side:** Starts at 0 for `ResponseHeadersEvent`, then increments for each `ResponseProgressEvent`, `ResponseCompleteEvent`, and `ResponseDoneEvent`.

These counters are NOT globally monotonic across the request/response boundary. They define ordering within their own phase.

### Helper Functions

`new_request_id() -> str`: Generates a UUID4 hex string for correlating events within a single API call.

`event_envelope(*, request_id, seq, provider, recv_ns=None) -> dict`: Builds the canonical envelope dict for constructing pipeline events. If `recv_ns` is None, stamps `time.monotonic_ns()` at call time.

`sse_progress_payload(sse_event) -> dict | None`: Maps a typed `SSEEvent` to the progress payload fields for `ResponseProgressEvent`. Returns None for SSE events that don't carry user-visible information (e.g., `TextBlockStartEvent`, `ContentBlockStopEvent`, `MessageStopEvent`).

## Event Lifecycle of a Single Streaming API Call

This is the canonical sequence for a typical Claude Code request that streams its response.

### 1. Request Phase (proxy thread)

The proxy receives an HTTP POST from Claude Code, parses the JSON body, and emits two events into the shared `event_queue`:

```
RequestHeadersEvent  (seq=0)   -- filtered HTTP headers
RequestBodyEvent     (seq=1)   -- parsed JSON body (messages, tools, system, etc.)
```

Both events share the same `request_id`. Headers are emitted first so the TUI can display them before the potentially large body.

**Excluded headers** (never emitted): `authorization`, `x-api-key`, `cookie`, `set-cookie`, `host`, `content-length`, `transfer-encoding`. Defined in `_EXCLUDED_HEADERS` frozenset in `proxy.py`.

**Gate condition:** Request events are only emitted for requests whose path matches the provider's `api_paths` AND whose body parses as a valid JSON object. Non-API traffic (health checks, token counting endpoints) is forwarded to the client but produces no pipeline events (though `LogEvent` is still emitted for all HTTP requests via `log_message()`). The path check is performed by `_expects_json_body()`, which tests against the provider's `ProviderSpec.api_paths` tuple:

- Anthropic: `("/v1/messages",)`
- OpenAI: `("/v1/chat/completions", "/chat/completions")`
- Copilot: `("/chat/completions", "/v1/chat/completions")`

If the body is present but fails JSON parsing or is not a top-level object, a warning is logged and no request events are emitted (the request is still forwarded).

### 2. Request Pipeline Processing

If a `RequestPipeline` is configured, it processes the parsed body:

1. **Transforms** run unconditionally in sequence, each seeing the output of the previous. A transform receives `(body, url)` and returns `(body, url)`.
2. **Interceptors** run after transforms. The first interceptor to return a non-None string short-circuits: the proxy generates a synthetic SSE response instead of forwarding to upstream.

If no interceptor fires, the body is re-serialized to bytes for upstream forwarding.

### 3. Upstream Forwarding and Response Headers

The proxy forwards the request to the upstream API. When the response arrives:

```
ResponseHeadersEvent (seq=0)   -- status code + filtered response headers
```

The proxy checks `Content-Type: text/event-stream` to determine the streaming vs. non-streaming path. The same `_EXCLUDED_HEADERS` set is applied to response headers.

### 4a. Streaming Response (typical path)

The SSE stream is processed by `_fan_out_sse`, which drives three sinks in parallel:

| Sink | Class | Role |
|------|-------|------|
| Client | `ClientSink` | Forwards raw bytes to Claude Code (transparent proxying). `on_raw` only. |
| Events | `EventQueueSink` | Parses SSE events, extracts progress hints, emits `ResponseProgressEvent`. |
| Assembler | `ResponseAssembler` or `OpenAiChatResponseAssembler` | Accumulates SSE fragments, reconstructs complete response on `on_done()`. |

All three sinks implement the `StreamSink` protocol (`on_raw`, `on_event`, `on_done`). Each sink call is error-isolated — a failure in one sink does not prevent others from receiving the event (logged via `_safe_sink_call`).

**SSE line parsing in `_fan_out_sse`:**
1. Each raw line is forwarded to all sinks via `on_raw()`.
2. Lines that don't start with `data: ` are skipped for event parsing.
3. `data: [DONE]` terminates the stream.
4. Other `data:` lines are JSON-parsed. If parsing fails, the line is silently skipped (raw bytes were already forwarded to the client).
5. The parsed event dict is dispatched to all sinks via `on_event(event_type, event)`, where `event_type` comes from `event.get("type", "")`.
6. After the stream ends (or on exception), `on_done()` is called on all sinks in a `finally` block.

**Progress extraction** during streaming produces `ResponseProgressEvent` for SSE events that carry user-visible information:

| SSE Event | Progress Fields Extracted |
|-----------|-------------------------|
| `message_start` | `model`, `input_tokens`, `cache_read_input_tokens`, `cache_creation_input_tokens` |
| `content_block_delta` (text) | `delta_text` |
| `message_delta` | `output_tokens`, `stop_reason` |
| `content_block_start` (tool_use, name="Task") | `task_tool_use_id` |

Each `ResponseProgressEvent` increments the response-side `seq` counter (starting from 0, so the first progress event has seq=1).

Not all SSE events produce progress events. `content_block_start` (text), `content_block_stop`, `content_block_delta` (input_json), and `message_stop` are silently consumed by the assembler but emit no progress hint.

**Provider-specific progress extraction:** The `EventQueueSink` selects an extraction strategy by protocol family via `_PROGRESS_EXTRACTORS_BY_FAMILY`:
- `"anthropic"` family: Parses via `parse_sse_event()` then calls `sse_progress_payload()`.
- `"openai"` family: Extracts from raw dicts directly (`_extract_openai_chat_progress`). The OpenAI extractor checks three candidate payloads in priority order: delta text content, finish reason, then model — returning the first non-None match.
- Unknown families fall back to the OpenAI extractor.

### 4b. Non-Streaming Response (alternative path)

For non-streaming responses (Content-Type is NOT `text/event-stream`), the proxy reads the entire body, writes it to the client, then emits:

```
ResponseHeadersEvent    (seq=0)  -- status + headers
ResponseCompleteEvent   (seq=1)  -- parsed response body (best-effort JSON decode)
```

**No `ResponseProgressEvent` or `ResponseDoneEvent` is emitted for this path.** This is confirmed in the code: after `self.wfile.write(data)`, the proxy emits only `ResponseHeadersEvent` and `ResponseCompleteEvent`, then returns.

The response body is decoded via `proxy_flow.decode_json_response_body()`, which returns `{}` on JSON parse failure or non-object responses.

### 5. Stream Completion (streaming path only)

After `_fan_out_sse` returns and all sinks have received `on_done()`, the proxy emits the finalization events:

```
ResponseCompleteEvent  (seq=N)    -- fully reconstructed message
ResponseDoneEvent      (seq=N+1)  -- terminal signal
```

The `seq` values continue from the last `ResponseProgressEvent`'s seq. If no progress events were emitted, the complete event gets seq=1 and done gets seq=2 (after the response headers at seq=0).

`ResponseCompleteEvent.body` is the assembler's `result` property. For Anthropic, this is a `ReconstructedMessage` with the same shape as a `stream=false` API response: `{id, type, role, content, model, stop_reason, stop_sequence, usage}`. For OpenAI, it matches the chat completion response shape: `{id, object, model, choices, usage}`.

If the assembler's `result` is None (no SSE events were received), `ResponseCompleteEvent` is NOT emitted — only `ResponseDoneEvent` is emitted.

`ResponseDoneEvent` is always the last event for a request_id in the streaming path. It signals that all processing for this API call is finished.

### 6. Error Paths

**HTTP error from upstream** (e.g., 429 rate limit, 500 server error — caught as `urllib.error.HTTPError`):

```
ErrorEvent (code=429, reason="Too Many Requests")
```

No `ResponseDoneEvent` follows. The error is terminal for this request_id. The `ErrorEvent` gets seq=0. The error response body and headers are still forwarded to Claude Code.

**Proxy-level failure** (connection refused, timeout, DNS failure — caught as generic `Exception`):

```
ProxyErrorEvent (error="Connection refused")
```

No `ResponseDoneEvent` follows. The `ProxyErrorEvent` gets seq=0. The proxy sends an HTTP 502 response to the client.

**Target resolution failure** (no target configured for reverse proxy mode):

```
ErrorEvent (code=500, reason="No target_host configured for reverse proxy mode")
```

This is emitted before any upstream connection attempt. The proxy responds with HTTP 500.

**CONNECT origin mismatch** (request URL origin does not match CONNECT tunnel origin):

```
ErrorEvent (code=403, reason="Origin mismatch: ...")
```

This is separate from the CONNECT host allowlist check (which happens during tunnel establishment). The origin constraint (`_constrain_target_origin` in `proxy_flow.py`) fires during request processing within an established tunnel, producing a 403 if the request targets a different origin than the one negotiated in the CONNECT handshake.

In all error cases, pipeline events (including errors) are only emitted if request events were also emitted for this request_id (i.e., the request had a valid JSON body on an API path). The `emitted_request` flag gates this.

**Exception:** Target resolution errors (`ErrorEvent` with status 500) are emitted unconditionally -- they fire before the body is parsed and before `emitted_request` is set. The `emitted_request` gate only applies to upstream HTTP errors and proxy errors.

## Complete Event Sequence Example

A typical Claude Code turn where the assistant responds with text:

```
                                         seq   recv_ns
RequestHeadersEvent  {headers: {...}}     0     t0
RequestBodyEvent     {body: {...}}        1     t1
ResponseHeadersEvent {200, headers}       0     t2
ResponseProgressEvent {model: "claude-opus-4-20250514", input_tokens: 12400, ...}
                                          1     t3
ResponseProgressEvent {delta_text: "I"}   2     t4
ResponseProgressEvent {delta_text: "'ll"} 3     t5
ResponseProgressEvent {delta_text: " read"}
                                          4     t6
...
ResponseProgressEvent {output_tokens: 847, stop_reason: "end_turn"}
                                          N-2   t_end-2
ResponseCompleteEvent {body: <full message>}
                                          N-1   t_end-1
ResponseDoneEvent                         N     t_end
```

All events share the same `request_id`. Note the two independent seq counters: request-side (0, 1) and response-side (0, 1, 2, ..., N).

## Replay Path

HAR replay (`har_replayer.py`) produces the same pipeline event types as the live proxy, but with a simplified four-event sequence since there is no streaming:

```
RequestHeadersEvent   (seq=0)
RequestBodyEvent      (seq=1)
ResponseHeadersEvent  (seq=2)
ResponseCompleteEvent (seq=3)
```

**Key differences from the live path:**

1. No `ResponseProgressEvent` or `ResponseDoneEvent` is emitted during replay.
2. The replay path uses a **single continuous seq counter** (0 through 3), unlike the live path which has separate request-side and response-side counters.
3. `recv_ns` timestamps are generated at replay time via `event_envelope()`, not preserved from original capture.
4. The replayer does NOT call `parse_sse_event` — it constructs `ResponseCompleteEvent` directly from the complete message stored in the HAR file.
5. Each HAR entry gets its own `request_id` via `new_request_id()`.

This means consumers must handle both "streaming with progress" and "direct complete" event sequences. The shared finalization path is `ResponseCompleteEvent` — it is the authoritative event for response content regardless of transport.

## Event Distribution

### Router Architecture

Events flow through a single `queue.Queue[PipelineEvent]` from producers (proxy, replayer) to the `EventRouter`, which fans out to subscribers on a dedicated daemon thread.

```
Producer(s) --> Queue --> EventRouter thread --> Subscriber 1 (on_event)
                                            --> Subscriber 2 (on_event)
                                            --> Subscriber N (on_event)
```

The router thread polls the source queue with a 0.5-second timeout in a loop, checking `_stop` (a `threading.Event`) between polls.

### Subscriber Types

| Type | Delivery | Use Case |
|------|----------|----------|
| `QueueSubscriber` | Buffers into its own `queue.Queue` for async consumption | TUI (events are drained on the Textual thread) |
| `DirectSubscriber` | Calls a function inline on the router thread | HAR recorder, analytics store, tmux controller |

Both implement the `Subscriber` protocol: any object with `on_event(event: PipelineEvent) -> None`.

### Ordering Guarantees

- Events for a single `request_id` are produced in causal order by the proxy thread. Since the proxy handles one request at a time per connection, request/response events for the same `request_id` are always ordered.
- The router preserves insertion order: events are drained from the source queue in FIFO order and delivered to each subscriber in subscriber-registration order.
- All subscribers see the same event in the same order.
- A subscriber exception does not prevent other subscribers from receiving the event (logged via `logger.exception("subscriber error")` and swallowed by the router).
- `QueueSubscriber` consumers must drain their own queue to maintain ordering; the router does not block on subscriber delivery.

### Concurrent Requests

Multiple API calls can be in flight simultaneously (e.g., Claude Code making parallel tool calls). Events from different `request_id` values may interleave on the shared queue. Consumers must use `request_id` to correlate events belonging to the same API call.

## SSE Parse Boundary

`parse_sse_event(event_type, raw)` is the single enforcer for Anthropic SSE data validation. It is called at one production site:

- **`proxy.py`** — live SSE stream parsing (both real upstream responses and synthetic interceptor responses)

The replayer does NOT call `parse_sse_event` — it constructs `ResponseCompleteEvent` directly from HAR body data.

The parse boundary converts raw JSON dicts into typed `SSEEvent` subclasses. Unknown event types raise `ValueError`. The dispatch table (`_SSE_PARSERS`) maps SSE type strings to parser functions:

| SSE type string | Parser output |
|----------------|---------------|
| `"message_start"` | `MessageStartEvent` |
| `"content_block_start"` | `TextBlockStartEvent` or `ToolUseBlockStartEvent` (based on `content_block.type`) |
| `"content_block_delta"` | `TextDeltaEvent` or `InputJsonDeltaEvent` (based on `delta.type`) |
| `"content_block_stop"` | `ContentBlockStopEvent` |
| `"message_delta"` | `MessageDeltaEvent` |
| `"message_stop"` | `MessageStopEvent` |

**Defensive parsing:** All parsers use `_str()` and `_int()` helpers that coerce arbitrary objects to the expected type rather than raising on unexpected input. Missing keys get default values (empty string, 0). Unrecognized `stop_reason` values fall back to `StopReason.NONE`. Unrecognized `content_block.type` values fall back to `TextBlockStartEvent`. Unrecognized `delta.type` values fall back to `TextDeltaEvent`.

**Note on the docstring:** The docstring in `event_types.py` claims `parse_sse_event` is called from `har_replayer.py` — this is stale. The replayer does not use it.

## Response Assembly

### Anthropic Assembly (`ResponseAssembler`)

The `ResponseAssembler` reconstructs complete Anthropic API responses from SSE fragments. It implements the `StreamSink` protocol and participates in `_fan_out_sse` alongside the client and event sinks.

**Assembly process:**
1. `on_event()` accumulates each raw event dict into `_events` list.
2. `on_done()` calls `reconstruct_message_from_events()` to produce a `ReconstructedMessage`.
3. The result is available via the `.result` property (None if no events were received).

**Reconstruction dispatch table** (`_EVENT_RECONSTRUCTORS`):
- `message_start`: Extracts id, model, role, usage into the message dict.
- `content_block_start`: Creates a new text or tool_use content block.
- `content_block_delta`: Appends text to the current text block, or accumulates `_input_json_str` for tool_use blocks.
- `content_block_stop`: Finalizes tool_use blocks by JSON-parsing the accumulated `_input_json_str` into the `input` field (falls back to `{}` on parse failure). Clears `current_text_block`.
- `message_delta`: Sets `stop_reason`, `stop_sequence`, and merges usage delta.

The reconstructed message shape:

```python
{
    "id": "msg_...",
    "type": "message",
    "role": "assistant",
    "content": [
        {"type": "text", "text": "..."},
        {"type": "tool_use", "id": "toolu_...", "name": "Read", "input": {...}},
    ],
    "model": "claude-opus-4-20250514",
    "stop_reason": "end_turn",
    "stop_sequence": null,
    "usage": {
        "input_tokens": 12400,
        "output_tokens": 847,
        "cache_read_input_tokens": 11000,
        "cache_creation_input_tokens": 0,
    },
}
```

### OpenAI Assembly (`OpenAiChatResponseAssembler`)

The `OpenAiChatResponseAssembler` reconstructs OpenAI chat completion responses from streaming chunks. It follows the same `StreamSink` protocol.

**Assembly process:**
1. `on_event()` accumulates each raw chunk dict into `_chunks` list.
2. `on_done()` calls `_reconstruct_openai_chat_message()`.
3. Chunks are processed through fixed-order reducers: identity (id, model), usage, then per-choice merging (finish_reason, delta content, delta tool_calls).

The reconstructed message shape:

```python
{
    "id": "chatcmpl-...",
    "object": "chat.completion",
    "model": "gpt-4",
    "choices": [
        {
            "index": 0,
            "message": {"role": "assistant", "content": "...", "tool_calls": [...]},
            "finish_reason": "stop",
        }
    ],
    "usage": {"prompt_tokens": 0, "completion_tokens": 0, "total_tokens": 0},
}
```

### Bridge: `sse_event_to_dict()`

The `sse_event_to_dict()` function in `response_assembler.py` converts typed `SSEEvent` subclasses back to raw dict format. This bridges the typed event world to the raw dict world used by reconstruction. **Currently unused in production code** — no call sites exist in `src/` outside its own definition. It is tested in `tests/test_response_assembler.py` and was likely intended for subscribers that need to feed typed events into `reconstruct_message_from_events`, but the proxy assemblers receive raw dicts directly via `on_event()` and never go through the typed SSEEvent layer for reconstruction.

## Multi-Provider Support

The event system supports multiple API providers. The `provider` field on every pipeline event identifies the source. Three providers are registered:

| Provider Key | Display Name | Protocol Family | API Paths | Proxy Mode |
|-------------|-------------|----------------|-----------|------------|
| `anthropic` | Anthropic | `anthropic` | `/v1/messages` | reverse |
| `openai` | OpenAI | `openai` | `/v1/chat/completions`, `/chat/completions` | reverse |
| `copilot` | Copilot | `openai` | `/chat/completions`, `/v1/chat/completions` | forward |

Provider-specific behavior is isolated to three dispatch points:

- **Progress extraction:** `_PROGRESS_EXTRACTORS_BY_FAMILY` dispatches by protocol family. Unknown families fall back to the OpenAI extractor.
- **Response assembly:** `_ASSEMBLER_CLASSES_BY_FAMILY` selects `ResponseAssembler` (anthropic) or `OpenAiChatResponseAssembler` (openai). Unknown families fall back to the OpenAI assembler.
- **SSE parsing:** Anthropic SSE events use typed `parse_sse_event`; OpenAI events are handled directly from raw dicts by the OpenAI progress extractor and assembler.

All downstream consumers see the same `PipelineEvent` types regardless of provider. The `ResponseCompleteEvent.body` shape differs by provider family (Anthropic Messages format vs. OpenAI Chat Completions format).

## Sentinel Interceptor

The sentinel system (`sentinel.py`) is a request pipeline interceptor that detects `$$` prefixed user messages and short-circuits the API call with a synthetic response. When triggered:

1. The request events (`RequestHeadersEvent`, `RequestBodyEvent`) are emitted normally (they were emitted before the pipeline ran).
2. `_send_synthetic_response()` generates synthetic SSE bytes via `_build_synthetic_sse_bytes()`.
3. The synthetic bytes are written directly to the client (not via `_fan_out_sse`).
4. The proxy then manually parses its own synthetic SSE bytes, feeding them through `parse_sse_event()` and a local `ResponseAssembler`.
5. Pipeline events are emitted in the same sequence as a real streaming response: `ResponseHeadersEvent` -> `ResponseProgressEvent`(s) -> `ResponseCompleteEvent` -> `ResponseDoneEvent`.

**Synthetic response details:**
- HTTP status is always 200 with `Content-Type: text/event-stream`.
- The model field comes from the request body's `model` field (falls back to `"synthetic"`).
- The response text is always `"[cc-dump]"` regardless of the command after `$$`.
- The SSE sequence is: `message_start` -> `content_block_start` (text) -> `content_block_delta` (text) -> `content_block_stop` -> `message_delta` (end_turn, 1 output token) -> `message_stop` -> `[DONE]`.

**Sentinel detection** (`extract_sentinel_command()`):
- Only checks the last message in the array. If its role is not `"user"`, detection does not fire -- it does not search backwards for a user message.
- Handles both string content (`"$$command"`) and block-list content (`[{"type": "text", "text": "$$command"}]`).
- For block-list content, only the first text block is checked.
- If the sentinel fires and a `TmuxController` is available, it focuses the cc-dump tmux pane.

This ensures the TUI, HAR recorder, and analytics all see a consistent event sequence even for intercepted requests.

## Dead Code

Two pipeline event types are defined but never constructed by any producer:

1. **`ResponseSSEEvent`** (`RESPONSE_EVENT`): Wraps an `SSEEvent` in a pipeline event. The TUI has a handler that translates it into `ResponseProgressEvent` via `sse_progress_payload()`, but this handler is never reached in practice because no code constructs `ResponseSSEEvent`. The proxy uses `EventQueueSink` to emit `ResponseProgressEvent` directly.

2. **`ResponseNonStreamingEvent`** (`RESPONSE_NON_STREAMING`): Carries a complete non-streaming HTTP response. The TUI has a handler that normalizes it into the complete-response path, but the proxy emits `ResponseCompleteEvent` directly for non-streaming responses instead.

Both types have corresponding `PipelineEventKind` enum values and handler registrations in the TUI dispatch table, but grep across the entire `src/` tree confirms no call site ever constructs an instance of either type.

## Gaps and Open Questions

1. **Forward proxy CONNECT flow:** When the proxy operates in forward-proxy mode (Copilot provider), CONNECT tunneling with TLS interception adds another layer. The decrypted requests flow through the same `_proxy()` method, with `_connect_target_host` constraining the upstream origin. The event sequence is identical to reverse proxy mode. However, TLS handshake failures produce no pipeline events — they are logged at DEBUG level and the connection is silently dropped.

2. **`LogEvent` lacks `request_id`:** `LogEvent` is emitted via `log_message()` which doesn't have access to the request_id. The `request_id` defaults to `""`, making it impossible to correlate `LogEvent` with other events for the same HTTP request.

3. **`do_OPTIONS` emits only `LogEvent`:** OPTIONS requests (CORS preflight) are handled directly with a fixed 200 response and CORS headers, without going through `_proxy()`. However, `send_response()` triggers `log_request()` -> `log_message()`, so a `LogEvent` is still emitted.

4. **`do_GET` follows the same path as `do_POST`:** GET requests go through the full `_proxy()` flow, but with no body they will never match API paths (since `body_bytes` will be empty), so they produce only `LogEvent`.
